================================================================================
CATEGORY 1: HOUSEHOLD REPAIR & MAINTENANCE
================================================================================

--- 1.1 ELECTRICIAN ---
INPUT:  job_type (fan/switch/wiring/socket/MCB), location, urgency
ALGORITHM:
  1. User selects job_type from menu → system maps to complexity tier (T1/T2/T3)
  2. base_price = price_table[job_type][city_tier]
  3. visit_fee  = flat Rs. 50-100 added always
  4. IF urgency == EMERGENCY: surge = 1.5x
  5. Geo-query electricians within 3km with verified_license == True
  6. Rank by trust_score + availability
  7. Confirm booking → send OTP to both
  8. Electrician enters start-OTP on arrival; end-OTP on completion
  9. Parts used: electrician logs each part → auto-adds to invoice
 10. Payment: UPI or cash; digital invoice with part breakdown

--- 1.2 PLUMBER ---
INPUT:  issue_type (leak/tap/commode/geyser/pipe_burst), photo_of_problem, location
ALGORITHM:
  1. User selects issue_type; prompt to upload photo
  2. Photo sent to matched plumber for remote quote BEFORE booking
  3. Plumber responds with quote within 15 min; user accepts/rejects
  4. IF user accepts: slot booked; deposit = 20% of quote
  5. On-site: if additional work found, plumber requests quote revision via app
  6. User approves revision OR disputes → escalate to platform agent
  7. End-OTP confirms job; balance payment released from escrow

--- 1.3 CARPENTER ---
INPUT:  furniture_type, problem_description, photo, dimensions (optional)
ALGORITHM:
  1. User uploads photo + describes issue (broken joint, hinge, polish)
  2. System suggests: repair vs. full-refurbish option with price difference
  3. Carpenter views photo, sends per-hour or per-job quote
  4. IF per-job: fixed price locked at booking
     IF per-hour: clock starts at start-OTP, stops at end-OTP
  5. Wood/material cost added separately; carpenter photographs materials bought
  6. Completion photo uploaded → user confirms → payment released

--- 1.4 PAINTER ---
INPUT:  room_count, room_dimensions (L x W x H), paint_type, surface_condition
ALGORITHM:
  1. User enters dimensions → system calculates total sq.ft.
  2. Paint type selection: distemper / emulsion / enamel / texture
  3. surface_condition: new plaster / old repaint / damaged (adds labour)
  4. auto_estimate = sq_ft * rate_per_sqft[paint_type] + primer_cost
  5. Colour visualisation tool: user picks shade → rendered preview
  6. Painter confirms estimate or adjusts after site visit
  7. Work split into days; each day requires end-of-day photo upload
  8. Final payment on completion photo approval

--- 1.5 MASON ---
INPUT:  job_description, photos, location, urgency
ALGORITHM:
  1. User posts job: tile repair / plaster patch / wall crack / boundary wall
  2. Platform opens bidding for 2 hours; local masons submit quotes
  3. Bids ranked by: trust_score * 0.5 + price_competitiveness * 0.5
  4. User picks from top 3 bids; advance = 30%
  5. Material procurement: mason logs cement/sand/tile purchase with photos
  6. Daily progress photos mandatory for jobs > 1 day
  7. Punch-list: user marks incomplete items before final payment

--- 1.6 WELDER ---
INPUT:  item_type (gate/grill/railing/tank), photos, material_preference
ALGORITHM:
  1. User uploads photos of item + specifies new fabrication vs. repair
  2. Welder with portable kit geo-matched within 5km
  3. Welder quotes: material cost (iron/SS) + labour separately
  4. IF fabrication: design approval photo before cutting starts
  5. Site visit for measurement; quote revision if dimensions change
  6. Completion: structural integrity photo + user sign-off via app

--- 1.7 AC REPAIR & SERVICING ---
INPUT:  brand, model, symptom (not_cooling/noise/leaking/no_start), service_type
ALGORITHM:
  1. service_type = annual_service | repair | gas_refill | installation
  2. IF annual_service: subscription model → auto-reminder every 6 months
  3. IF repair: symptom checklist narrows to fault category
     fault_map = {not_cooling → gas/coil, noise → fan/motor, leaking → drain}
  4. Technician verifies OEM authorisation for premium brands
  5. Spare part needed? → technician checks availability; user approves part cost
  6. OTP start/stop; gas refill quantity logged in app
  7. Subscription AMC: track service history per unit per household

--- 1.8 WASHING MACHINE REPAIR ---
INPUT:  brand, model_number, symptom, machine_type (top/front_load)
ALGORITHM:
  1. Brand + model lookup → known fault database suggests likely cause
  2. Symptom checklist: not_spinning / leaking / error_code / not_heating
  3. IF error_code: user types code → system maps to fault + estimated cost
  4. Technician confirmed as OEM-authorised or multi-brand certified
  5. Spare part pre-check: technician carries common spares for top brands
  6. Repair completed → test cycle run → user confirms before payment

--- 1.9 MICROWAVE / INDUCTION REPAIR ---
INPUT:  appliance_type, brand, symptom
ALGORITHM:
  1. Appliance type determines technician skill-set filter
  2. Self-diagnosis tree shown to user BEFORE booking:
     not_heating → magnetron | turntable_not_rotating → motor | sparks → coating
  3. IF self-diagnosis resolves issue (e.g. child lock): no booking needed → free
  4. Else: book technician; bring appliance to workshop OR home visit option
  5. Workshop option: pickup + drop included; 3-day TAT with progress SMS

--- 1.10 TV REPAIR ---
INPUT:  tv_type (LED/LCD/OLED), issue_category, brand, screen_size
ALGORITHM:
  1. Issue pre-classification:
     set_top_box_issue vs. display_issue vs. remote_issue vs. no_signal
  2. Remote issue → platform sends replacement directly; no technician needed
  3. No signal → app runs signal diagnostic checklist; may resolve without visit
  4. Display/board issue → technician visit; screen replacement quote upfront
  5. Carry-in option for large repairs; doorstep for minor issues
  6. Warranty check: system flags if device is under manufacturer warranty

--- 1.11 WATER PURIFIER SERVICING ---
INPUT:  brand, model, last_service_date, service_type (filter_change/membrane/full)
ALGORITHM:
  1. AMC model: user signs up → platform tracks filter life based on usage days
  2. Auto-reminder at 85% filter life; push notification to user
  3. Technician visits with pre-packed filter kit for that model
  4. Service log updated: date, technician ID, parts replaced, TDS reading before/after
  5. Pay-per-service: user books on demand; no subscription required
  6. Filter disposed of by technician; eco-disposal certificate generated

--- 1.12 GAS STOVE & CHIMNEY REPAIR ---
INPUT:  appliance (gas_stove/chimney), issue_type, burner_count (stove)
ALGORITHM:
  1. Gas stove: issue menu → burner_not_lighting / regulator_leak / knob_broken
     Safety check: IF regulator_leak → flag as EMERGENCY; surge + priority dispatch
  2. Chimney: issue menu → not_sucking / noisy / oil_dripping / filter_clogged
  3. Chimney deep-clean listed as fixed-price SKU; no quote needed
  4. Technician carries standard nozzles, igniters, regulator stock
  5. Post-repair: flame test photo uploaded for gas stove jobs
  6. Same-day guarantee for gas stove emergencies; premium surcharge applied

--- 1.13 INVERTER & UPS SERVICING ---
INPUT:  brand, battery_count, last_service_date, issue (not_charging/low_backup/alarm)
ALGORITHM:
  1. Scheduled maintenance: quarterly reminder auto-generated on registration
  2. Services: battery_water_topup | terminal_cleaning | load_test | replacement
  3. Battery water top-up → fixed low-cost SKU; 15-min job
  4. Load test result logged: backup_hours measured → recommendation if < threshold
  5. Battery replacement: old battery buyback amount shown; deducted from new cost
  6. UPS for computers: downtime minimised; technician brings replacement unit

--- 1.14 MOSQUITO NET INSTALLATION ---
INPUT:  window_count, window_dimensions, material (fibreglass/aluminium/SS), frame_type
ALGORITHM:
  1. Guided AR measurement: user points camera at window → app auto-calculates W x H
  2. Material selection with price difference shown
  3. Frame type: fixed / sliding / magnetic / velcro
  4. Quote = (area * material_rate) + (window_count * installation_fee)
  5. Fabrication TAT: 2-3 days; installation on delivery day
  6. Post-install: each window photographed; user confirms fit before payment

--- 1.15 GLASS PANE REPLACEMENT ---
INPUT:  glass_type, exact_dimensions (L x W), thickness, location (window/door/table)
ALGORITHM:
  1. User enters dimensions + uploads photo of frame with reference object for scale
  2. Thickness options: 4mm / 5mm / 8mm / toughened; price auto-calculated
  3. Platform queries nearby glass shops for stock availability
  4. Fastest-available shop selected; cutting + delivery + installation bundled
  5. Fragile item insurance: Rs. 50 add-on covers breakage during transit
  6. Installation: old glass removed safely; disposal fee included in quote

CATEGORY 2: PERSONAL CARE & GROOMING
================================================================================

--- 2.1 LADIES' PARLOUR AT HOME ---
INPUT:  services_needed (threading/waxing/facial/cleanup), skin_type, product_brand_pref
ALGORITHM:
  1. Service multi-select with individual prices shown; bundle discount auto-applied
  2. Beautician profile includes: skill tags, portfolio photos, product brands carried
  3. User can specify brand preference (Lotus / VLCC / Shahnaz / Gold)
  4. Slot booking: 60-min blocks; system calculates duration from services selected
  5. Safety: beautician shares live location from 30 min before slot
  6. Post-service: before/after photos (with consent) → portfolio; rating collected

--- 2.2 HOME-VISIT BEAUTICIAN (BRIDAL / PARTY) ---
INPUT:  event_type, event_date, services (makeup/saree_draping/hair), trial_needed
ALGORITHM:
  1. Event date selected → platform shows available bridal specialists
  2. Portfolio gallery: real client photos, not stock images; verified tag
  3. Trial session booking: 50% of final makeup price, adjustable to full booking
  4. Video call consultation: discuss skin tone, lehenga colour, reference photos
  5. Package builder: makeup + hair + saree draping + touch-up kit = bundle price
  6. Advance payment: 40% on booking; balance on event day after satisfaction check
  7. Cancellation policy: > 7 days = full refund; < 3 days = 50% forfeiture

--- 2.3 MEN'S SALON AT HOME ---
INPUT:  services (haircut/beard_trim/shave/head_massage/facial), barber_preference
ALGORITHM:
  1. Service selection with duration estimate shown
  2. Geo-tagged barbers within 2km; rating + photo + years_experience displayed
  3. "Express haircut" (20 min, basic) vs. "Full grooming" (60 min, all services)
  4. Barber carries sterilised kit (confirmed via kit-photo upload before visit)
  5. Hair style selection: user uploads reference photo; barber confirms feasibility
  6. Payment after service; no advance for single visits

--- 2.4 MEHENDI ARTIST BOOKING ---
INPUT:  design_style (Arabic/Rajasthani/Bridal/Indo-Arabic), coverage, event_date
ALGORITHM:
  1. Design catalogue: artist uploads 20+ designs per style category
  2. Coverage pricing: per_hand / per_feet / full_arm / bridal_full (both hands+feet)
  3. Festive season (Eid, Karva Chauth, Teej): advance booking opens 30 days early
  4. Custom design: user uploads reference image; artist quotes separately
  5. Duration estimate shown per coverage type
  6. Post-mehendi: drying time advisory sent; artist leaves drying cone provided

--- 2.5 SPA / MASSAGE AT HOME ---
INPUT:  massage_type (relaxation/Ayurvedic/deep_tissue/sports), duration, oil_type
ALGORITHM:
  1. Therapist must hold certification (VLCC / Ayurvedic diploma / NSDC); verified badge
  2. Gender filter: user selects therapist gender (mandatory safety feature)
  3. Oil type: coconut / sesame / lavender / Ayurvedic herbal; displayed with benefits
  4. Duration options: 30 min / 60 min / 90 min; price scales linearly
  5. Safety protocol:
     - Therapist shares live location to emergency contact of user
     - Mandatory OTP sharing: user's friend/family receives booking details
     - SOS button active during entire session
  6. Post-session: hydration advisory; follow-up session discount offered

--- 2.6 EAR PIERCING ---
INPUT:  age_group (infant/child/adult), piercing_type, material (gold/silver/steel)
ALGORITHM:
  1. Age group determines equipment: gun (adult) vs. needle (infant/sensitive)
  2. Professional carries sterilisation certificate (shown in profile)
  3. Material selection: hypoallergenic options flagged for sensitive skin
  4. For infants: parent consent confirmation required in app (checkbox + OTP)
  5. After-care instructions delivered via WhatsApp immediately after service
  6. 7-day follow-up check ping: user reports any infection; escalate to doctor if needed

CATEGORY 3: FOOD & BEVERAGE
================================================================================

--- 3.1 TIFFIN SERVICE ---
INPUT:  meal_type (veg/non-veg), roti_count, subscription_period, delivery_time
ALGORITHM:
  1. Home chef registers: uploads daily menu, availability calendar, kitchen photo
  2. FSSAI certificate check (mandatory for commercial kitchens; advisory for home)
  3. Customer subscribes: weekly / monthly; choose roti count + extras
  4. Auto-debit via UPI AutoPay mandate on subscription start
  5. Daily menu published by chef by 8am; customer notified
  6. Pause feature: customer can pause up to 5 days/month; credits carry forward
  7. Delivery: chef's own delivery OR integrated with local delivery boy pool
  8. Chef ledger: auto-calculates daily earnings; GST invoice if above threshold

--- 3.2 LOCAL MESS / FOOD PLATE ---
INPUT:  meal_time (lunch/dinner), date, portion_size, pickup_or_delivery
ALGORITHM:
  1. Mess owner publishes menu for each meal slot by 9am (lunch) / 4pm (dinner)
  2. Fixed plates available: limited quantity; counter decrements on each booking
  3. IF quantity == 0: show "sold out"; offer nearby mess alternative
  4. Booking generates token number with QR code
  5. Pickup: user scans QR at mess counter; contactless handoff
  6. Delivery: token linked to delivery boy; GPS tracking to doorstep
  7. Rate: per-plate price locked at booking; no surge for advance bookings

--- 3.3 CATERING (SMALL FUNCTIONS) ---
INPUT:  event_type, guest_count, cuisine, service_type (buffet/sit-down), staff_needed
ALGORITHM:
  1. Package builder:
     a. Select dishes per category (starter/main/dessert) from caterer's menu
     b. System calculates per-plate cost = dish_costs / guest_count + overhead
  2. Staff count auto-suggested: 1 server per 25 guests (buffet) / 15 (sit-down)
  3. Real-time quote generated; caterer confirms within 2 hours
  4. Advance payment: 30% on booking; balance 24hrs before event
  5. Logistics checklist pushed to caterer: utensils, gas cylinders, staff list
  6. On event day: caterer checks in via app; guest count confirmed for final billing
  7. Leftover food management: NGO donation option shown post-event

--- 3.4 HOME DAIRY DELIVERY ---
INPUT:  milk_type (cow/buffalo/mixed), quantity_litres, delivery_slot (morning/evening)
ALGORITHM:
  1. Milkman registers: dairy license, cattle count, milk quality certificate
  2. Customer subscribes: quantity + slot; UPI AutoPay mandate setup
  3. Daily: customer can alter quantity via WhatsApp chatbot by 9pm for next morning
  4. Pause: tap-to-pause up to 7 days; ledger auto-adjusts
  5. Digital ledger: running balance visible to both customer and milkman
  6. Monthly settlement: auto-calculated; UPI collect sent to customer
  7. Quality complaint: customer rates each delivery; 3 consecutive low ratings → review
  8. Holiday calendar: milkman marks unavailable dates; customer notified in advance

--- 3.5 HOME BAKER ---
INPUT:  product_type (cake/cookies/bread/pastry), customisation, quantity, delivery_date
ALGORITHM:
  1. Baker's Instagram-style storefront: product photos, price per unit/kg, lead time
  2. Custom cake order:
     a. User uploads reference photo
     b. Selects: flavour / eggless / fondant vs. cream / size (kg)
     c. Personalised message (max 30 chars)
  3. Baker confirms feasibility within 4 hours; sends design sketch photo
  4. Advance: 50% on confirmation; balance on delivery
  5. Delivery date must be ≥ baker's stated lead time (2-5 days)
  6. Packaging: baker selects box type; fragile label auto-generated
  7. Delivery: baker's own or platform logistics; temperature-advisory attached

--- 3.6 SWEET SHOP / HALWAI CUSTOM ORDERS ---
INPUT:  occasion, sweets_list, quantity_kg_per_item, packaging_type, delivery_date
ALGORITHM:
  1. Festive / wedding order flow:
     a. Item list with per-kg price; bulk discount auto-applied above 5kg
     b. Box packing: choose box size, colour, ribbon, personalised message
  2. Order minimum: Rs. 500 for delivery orders
  3. Advance: 40% for orders > Rs. 2000
  4. Production tracking: halwai marks "in preparation" → "packed" → "out for delivery"
  5. Shelf-life advisory: sent with order (consume within X days)
  6. Bulk wedding orders: multiple delivery slots across event days; coordinator assigned

--- 3.7 NAMKEEN / SNACK MAKER BULK ORDER ---
INPUT:  items_selected, quantity_per_item_kg, delivery_address, occasion
ALGORITHM:
  1. Festive combo pack builder: pre-set combos OR custom item + quantity selection
  2. Price updates dynamically as items are added
  3. Minimum order for delivery: 2 kg aggregate
  4. Packing: vacuum-sealed or tin (option); price difference shown
  5. Freshness guarantee: made-to-order; production starts only after payment
  6. Production TAT: 1-3 days based on quantity
  7. Delivery: local courier or self-pickup with 10% discount for pickup

CATEGORY 4: TRANSPORTATION & MOVING
================================================================================

--- 4.1 SHARED AUTO / E-RICKSHAW / GRAMIN SEWA ---
INPUT:  route_id OR origin+destination, seat_count, travel_date_time
ALGORITHM:
  1. Route registry: all fixed routes in district registered with stops, fare, frequency
  2. User selects route → sees live vehicle location + seat availability
  3. Seat booking: reserve 1-4 seats; digital pass generated (QR)
  4. Pass types: single / daily / weekly / monthly; pricing tiered
  5. Driver confirms boarding via QR scan; seat count decremented
  6. Missed-call booking: call route number → IVR confirms next available seat
  7. Payment: UPI on app; cash to driver with digital receipt via SMS
  8. Live ETA: updated every 2 min; push notification when vehicle is 5 min away

--- 4.2 SMALL GOODS SHIFTING (TEMPO / PICKUP) ---
INPUT:  pickup_location, drop_location, item_description, item_photos, floor_info
ALGORITHM:
  1. User enters pickup + drop; system calculates distance
  2. Item type selection: furniture / boxes / appliances / mixed
  3. Floor info: pickup_floor + drop_floor; staircase vs. lift; labour charge added
  4. Photo upload of load: driver reviews before accepting to avoid disputes
  5. Instant fare = base_fare + (distance * per_km_rate) + labour_charges
  6. Multiple nearby vehicle owners shown: truck / tempo / pickup sorted by capacity
  7. GPS tracking link shared with user on booking confirmation
  8. Loading photo + unloading photo mandatory; damage claim window = 24 hrs post-delivery

--- 4.3 BIKE / CAR WASH AT DOORSTEP ---
INPUT:  vehicle_type, wash_type (basic/foam/full_detail), add_ons, time_slot
ALGORITHM:
  1. Vehicle type determines base price: two-wheeler / sedan / SUV / van
  2. Wash type menu: basic rinse / foam wash / interior vacuum / full detailing
  3. Add-ons: wax polish / tyre dressing / engine cleaning (each priced separately)
  4. Slot booking: 30-min or 60-min window depending on wash type
  5. Pre-wash photo: operator uploads before starting (documents pre-existing scratches)
  6. Post-wash photo: uploaded before marking complete; user approves via app
  7. Water usage: operator carries tank; eco-wash option uses 80% less water
  8. Subscription: weekly/monthly wash plans at discounted rate via UPI mandate

--- 4.4 TWO-WHEELER MECHANIC AT HOME ---
INPUT:  vehicle_make_model, symptom, urgency
ALGORITHM:
  1. Symptom selection tree:
     self_start_issue → battery/motor | chain_loose → adjustment
     tyre_flat → puncture_repair | brake_issue → pad/cable
     engine_noise → diagnostic required
  2. Common symptoms → fixed-price SKU shown immediately
  3. Diagnostic required: mechanic visits with OBD scanner for modern bikes
  4. Mechanic carries standard spare parts inventory (listed in profile)
  5. Part needed but not in stock: mechanic fetches from nearby shop; part photo + receipt uploaded
  6. Test ride: mechanic rides 200m; user confirms resolution before payment
  7. Warranty on labour: 7 days; on parts: as per part manufacturer

--- 4.5 CYCLE REPAIR AT HOME ---
INPUT:  issue_type (puncture/brake/gear/chain/saddle), cycle_type
ALGORITHM:
  1. Simple drop-down issue selection; flat visit fee + part cost transparent upfront
  2. Puncture repair: fixed price inclusive; no part cost ambiguity
  3. Gear tuning / brake adjustment: labour-only; no parts needed in most cases
  4. Repairman carries: tube patches, brake cables, chain links, lubricant
  5. Service completes in < 30 min typically; real-time updates not needed
  6. Kids' cycle: age-appropriate safety check included at no extra cost
  7. Rating after service; loyalty: 5th repair free on platform

--- 4.6 LPG CYLINDER DELIVERY BOOKING ---
INPUT:  agency_name, consumer_id, cylinder_type (14.2kg/5kg/commercial), refill_needed
ALGORITHM:
  1. User registers consumer ID + agency; platform integrates with agency's system
  2. One-tap "Book Refill" → request sent to agency with consumer ID
  3. Subsidy tracking: platform shows how many subsidised refills remain for the year
  4. Delivery slot: agency confirms date/time window (half-day slots)
  5. OTP-based delivery confirmation: delivery agent enters OTP; consumer confirms receipt
  6. Empty cylinder exchange: mandatory; app flags if no empty returned
  7. Emergency cylinder: platform connects to nearest stockist; higher price shown transparently
  8. Payment: already linked via BHIM/UPI at agency; or cash to delivery agent with digital receipt

CATEGORY 5: HEALTH & WELLNESS
================================================================================

--- 5.1 LOCAL RMP / INFORMAL DOCTOR CONSULTATION ---
INPUT:  symptoms, patient_age, language, urgency, consultation_type (audio/video)
ALGORITHM:
  1. Symptoms entered in vernacular voice or text
  2. Symptom triage: platform flags RED symptoms (chest pain, unconscious, bleeding)
     → redirect to emergency services; do NOT match RMP for RED flags
  3. For non-emergency: match RMP/compounder within locality
  4. Consultation via video/audio call in patient's language
  5. Prescription: generated in app (drug name, dosage, days); shared on WhatsApp
  6. Digital health record: maintained per patient; accessible in future consultations
  7. Fee: UPI after consultation; platform holds until patient confirms received care
  8. Referral: RMP can issue in-app referral to higher facility with record attached

--- 5.2 INJECTION / IV DRIP AT HOME ---
INPUT:  prescription_photo, drug_name, dosage, patient_details, urgency
ALGORITHM:
  1. MANDATORY: prescription upload verified before job is dispatched
  2. Drug validation: system cross-checks drug name against scheduled drug list
     → Schedule H drugs require doctor verification before nurse accepts
  3. Nurse/compounder accepts job; navigates to patient location
  4. On arrival: vein condition photo (IV) or injection site photo logged
  5. Drug administered: photo of ampoule/vial with batch number uploaded
  6. Digital record: drug name, batch, dose, time, administrator ID — immutable log
  7. Post-procedure: 30-min observation window; nurse confirms patient stable before leaving
  8. Follow-up alert: platform pings patient in 2 hours for wellbeing check

--- 5.3 WOUND DRESSING & STITCHING ---
INPUT:  wound_type, wound_photo, patient_age, tetanus_status
ALGORITHM:
  1. User uploads wound photo for preliminary assessment
  2. Platform nurse triages from photo: minor (dress at home) vs. needs_stitching vs. hospital_referral
  3. Nurse visits with sterile dressing kit; gloves + mask mandatory (photo proof on arrival)
  4. Dressing procedure logged: wound size, dressing type, next_change_date
  5. Stitching: only qualified nurse/compounder certified for minor suturing
  6. Auto-scheduled follow-up: next dressing change appointment booked immediately
  7. Tetanus status: if unknown → nurse advises and can administer TT injection (add-on service)
  8. Progress photos at each visit; healing tracked in timeline view

--- 5.4 PHYSIOTHERAPY / BONE SETTING AT HOME ---
INPUT:  ailment (back_pain/sprain/fracture_rehab/sports_injury), session_count, patient_age
ALGORITHM:
  1. Ailment selection → platform suggests session package (minimum 5 sessions for most conditions)
  2. Therapist profile: qualification, AIPT/IAP membership, experience years
  3. Initial assessment session (paid): therapist evaluates and creates treatment plan
  4. Session plan pushed to patient app with exercises between sessions
  5. Exercise videos (vernacular narration) sent via app for home practice
  6. Each session logged: techniques used, patient feedback, pain_score (1-10)
  7. Progress chart: pain_score tracked across sessions; visible to patient and therapist
  8. Bone setter (traditional): listed separately; folk_practitioner tag with disclaimer shown

--- 5.5 VETERINARY FIRST AID FOR CATTLE ---
INPUT:  animal_type, animal_count, symptoms, farm_location, urgency
ALGORITHM:
  1. Farmer uploads: symptom description + animal photo
  2. Symptom triage by platform vet (async within 30 min):
     bloat → emergency dispatch | fever → schedule visit | wound → nurse visit
  3. Nearby pashu doctor (govt. or private) matched by location
  4. Doctor navigates to farm using GPS; ETA shared with farmer
  5. Treatment record: drug administered, dosage, diagnosis, next visit date
  6. Digital payment to private vet; govt. vet fees tracked for subsidy reconciliation
  7. Herd management: farmer can register all animals; treatment history per animal maintained

--- 5.6 ANIMAL ARTIFICIAL INSEMINATION ---
INPUT:  cattle_breed, heat_cycle_date, animal_tag_id, semen_breed_preference
ALGORITHM:
  1. Heat cycle tracking: farmer logs onset → system calculates optimal insemination window (12-18 hrs)
  2. Alert sent to nearest AI technician with animal location + breed preference
  3. Technician accepts and navigates within the insemination window
  4. Procedure logged: technician ID, semen batch number, breed, timestamp
  5. Outcome tracking: 21-day heat cycle check reminder sent to farmer
  6. IF no heat signs at day 21 → mark as potentially pregnant; schedule vet confirmation
  7. Success record maintained per animal: conception rate history
  8. Semen inventory: platform aggregates demand → bulk procurement for technicians

--- 5.7 YOGA CLASS BOOKING ---
INPUT:  yoga_style (Hatha/Ashtanga/Pranayama/Yin), session_type (group/individual), location_pref
ALGORITHM:
  1. Trainer profile: style tags, certification body, years_teaching, location (park/rooftop/community_hall)
  2. Group class: fixed slots; seat counter; max group size set by trainer
  3. Individual: book any available slot; trainer confirms
  4. Location: trainer's location or user's home (home visit = premium)
  5. Package booking: 8-session / 16-session packages at discounted rate; non-refundable
  6. Live location shared by trainer 15 min before session start
  7. Session attendance: QR check-in at location; marks attendance for package tracking
  8. Substitute trainer: if original trainer cancels < 2 hrs → platform arranges substitute OR full refund

--- 5.8 HOME SAMPLE COLLECTION ---
